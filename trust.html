Trust · Threat model · Independent verification

What can be independently proven.

CEYO trust posture is based on deterministic mechanics and explicit boundaries. Verification is mechanical: canonicalize → hash → signature → policy alignment. It confirms integrity and provenance under policy scope — not correctness of model output.

Guarantees vs. non-guarantees

Clear statements of what CEYO proves and what it does not claim.

What CEYO can prove
  • Integrity: the artifact has not been modified since sealing.
  • Provenance: the signature validates against the declared public key reference.
  • Scope boundary: policy ID/version (and policy hash, where used) identifies what was captured.
  • Determinism: canonicalization produces reproducible bytes for independent recomputation.
What CEYO does not claim
  • It does not confirm decision correctness, fairness, or legality.
  • It does not certify regulatory compliance or admissibility.
  • It does not guarantee inference reproducibility.
  • It does not guarantee key custody integrity outside the operator’s environment.
Verification steps (preview) Deterministic procedure
  1. Canonicalize the artifact payload under the recorded canonicalization scheme (RFC 8785).
  2. Recompute the SHA-256 digest over canonical bytes.
  3. Validate the digital signature using the referenced public key.
  4. Confirm policy ID/version/hash alignment with the expected policy document.
Output states: PASS / FAIL / POLICY_MISMATCH. These states confirm integrity posture — not correctness of model behavior.

Threat model (simple)

Common integrity risks and how CEYO’s mechanics address them. Expand each for detail.

Tampering after capture Integrity

Modification of an artifact after it is generated.

Canonicalization + hashing + signature create tamper-evidence. Any post-seal changes break digest/signature validation and produce a FAIL outcome.
Replay of an artifact Provenance

Reusing a valid artifact in the wrong context or time window.

Artifacts embed timestamp and policy version identifiers. Operators can also include environment fingerprints and request IDs within policy scope to bind artifacts to a specific event context.
Selective omission Scope

Failing to generate artifacts for some events.

CEYO supports audit-ready logging of seal failures and generation status. Operators can couple artifact IDs with upstream request logs and retention policies to detect gaps.
Key compromise Custody

Signing key is compromised in the operator environment.

CEYO is non-custodial: keys stay with the operator (HSM/KMS/TEE). Mitigation depends on operator key governance: rotation, revocation, and incident handling. Verification remains possible with correct public key chain practices.
