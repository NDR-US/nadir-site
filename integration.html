DEPLOYMENT PATTERNS

Choose Your Boundary
Sidecar · Gateway · Wrapper

A / Sidecar
near runtime
Service-adjacent instrumentation
Deploy alongside inference. Best for microservices and hosted inference APIs where you want minimal coupling.
Pros: low intrusion · clear boundary · easy rollout
B / Gateway
edge boundary
Ingress/egress enforcement point
Wrap calls at a shared boundary (API gateway / proxy). Useful when policy and disclosure tiering are centralized.
Pros: centralized policy · multi-service coverage · consistent artifacts
C / Wrapper
library
SDK-level capture
Instrument within an app or pipeline via a library wrapper. Useful for research, batch jobs, and controlled internal environments.
Pros: fine-grained control · dev-friendly · simple tests

INTEGRATION STEPS

Policy → Capture → Canonicalize → Seal

01
Define capture policy (versioned)
Declare fields included/excluded/masked. Choose disclosure tier(s). Embed policy ID + version in every artifact.
02
Select canonicalization + sealing suites
Choose deterministic canonicalization and cryptographic suite aligned to institutional requirements. Record algorithms in the artifact.
03
Set key custody boundary
Keep private signing keys operator-side (HSM/KMS/TEE). Establish public key distribution and authorization path.
04
Run fail-open with explicit failure marking
If artifact sealing fails, inference continues. Log failure events. Mark artifacts as unsealed for review.
integration.notes
operator checklist
- Identify inference boundary (svc/gateway/sdk)
- Choose capture scope (minimize)
- Choose disclosure tier(s)
- Select canonicalization algorithm
- Select hash + signature suites
- Confirm key custody (HSM/KMS/TEE)
- Decide artifact storage + retention
- Define verification roles + access
- Validate fail-open behavior + logs
Engineering Boundary
CEYO is not an enforcement layer and does not decide whether capture choices are appropriate. Policy is institutional. CEYO makes scope explicit and verification reproducible.
VERIFICATION

Independent Verification Procedure
Conceptual

A verifier needs: the artifact, the declared policy, the canonicalization procedure, and authorized public key material. Verification recomputes integrity checks; it does not interpret correctness.
Controlled review includes the reference procedure and test vectors for your selected suite.
verify.sh
illustrative
1) Read artifact + policy identifiers
2) Canonicalize fields (RFC8785)
3) Compute SHA-256 digest
4) Compare digest to recorded value
5) Verify signature with public key
6) Confirm disclosure tier + scope
7) Record verification result (pass/fail)
Boundary Declaration · V2026.02.19

CEYO produces records. It does not validate correctness, fairness, legality, or compliance outcomes. Authorized institutions and operators retain responsibility for determinations.

Verification confirms integrity and provenance under policy scope — not correctness of model output. CEYO does not certify regulatory compliance and does not guarantee admissibility.
