[01] Privacy Principles

Five PCL Principles

CEYO's PCL Framework is organized around five principles that together constitute privacy-by-design in the evidentiary infrastructure context. Each principle maps directly to an architectural constraint.

01
Data Minimization
Only fields explicitly defined in capture policy are captured. No implicit collection. No ambient telemetry beyond declared scope. Minimization is enforced architecturally, not by policy review.
02
Purpose Limitation
Capture policy defines the purpose for which fields are included. Artifact fields reflect declared purpose. Post-hoc expansion of captured data is not possible — the sealed artifact reflects only what was captured.
03
Pre-Capture Masking
Masking rules are defined before capture begins. Sensitive fields are masked, redacted, or hashed before inclusion in the artifact. Masking occurs pre-capture — not post-collection redaction.
04
Tiered Disclosure
Disclosure scope is policy-defined and enforced at the tier level. Verification does not require exposure of sensitive fields. Disclosure boundaries are set before artifact generation, not after.
05
Non-Custodial Control
CEYO does not hold signing keys. Operator retains control. No data retention by CEYO beyond operational necessity. Privacy authority remains with deploying institution.
[02] Capture Policy Architecture

Field Disposition Categories

Every field in a CEYO artifact has an explicit disposition defined by the capture policy. There are four dispositions: Include, Exclude, Mask, and Conditional. No field has an implicit disposition.

DispositionDefinitionPrivacy ImplicationExample Fields
Include Field is captured in full and included in the artifact at its declared disclosure tier. Only fields where full capture is appropriate for the declared purpose. timestamp, policy_id, policy_version, env_fingerprint
Exclude Field is not captured. It is not present in the artifact. No representation or hash is recorded. Default disposition for all fields not explicitly configured. Model weights, training data, unrelated telemetry. Model weights, training datasets, proprietary architecture parameters
Mask Field is captured as a hash or redacted form. The original value is not preserved. The masked form supports integrity verification without exposing the original. Fields with privacy or confidentiality sensitivity where verification need can be met by hash reference. input_ref_hash, PII fields where configured, confidential context fields
Conditional Field capture depends on a policy-defined condition evaluated at capture time. Capture decision is deterministic and documented in policy version. Context-sensitive fields where capture appropriateness depends on inference context. Sector-specific context fields, jurisdiction-dependent fields
[03] Masking Architecture

Pre-Capture Masking

Masking in CEYO is architectural, not operational. It occurs before any field value enters the artifact pipeline. The following masking approaches are available.

Hash Reference
SHA-256 of Field Value
Field value is hashed before capture. Only the hash is included in the artifact. Original value is not captured. Supports integrity verification: if the original value is known, the hash can be independently validated.
Redaction
Field Present, Value Absent
The field key is present in the artifact schema but the value is replaced with a declared redaction marker. Confirms the field was considered in policy scope without exposing value. Reviewers can verify field existence without value exposure.
Tokenization
Reversible Reference Token
Field value replaced by a policy-defined token. Original value resolvable only by authorized parties with token map access. Supports limited disclosure under controlled conditions without exposing original in artifact.
Aggregate Binning
Value Range or Category
Specific value replaced by a policy-defined category or range. Reduces precision to the minimum required for the declared purpose. Appropriate for numerical fields where range information is sufficient for oversight purposes.
[04] Constrained Disclosure

Disclosure Tier Architecture

Disclosure tier enforcement is the mechanism by which CEYO supports verification without requiring full artifact exposure. Tier assignment is policy-defined before artifact generation. The following tiers are supported.

Public / Redacted
Artifact summary, policy identifier, non-sensitive metadata, and integrity hash references. No controlled-tier fields exposed. Available without executed controlled review agreement. Supports public verification of artifact existence and policy scope.
Controlled Review
Expanded contextual fields, integrity data, policy-aligned elements. Requires executed Controlled Review Agreement. Disclosure bounded by agreement terms. Verification of full integrity chain and policy alignment.
Institutional — On Request
Full artifact representation within the scope of applicable law and operator policy. Access evaluated per institution and proceeding context. May require court order, regulatory demand, or institutional authorization per applicable legal framework.
Boundary Declaration  ·  PCL Framework  ·  V2026.02
The CEYO PCL Framework defines architectural privacy properties. It does not constitute legal advice on privacy compliance in any jurisdiction. GDPR, CCPA, HIPAA, and other jurisdiction-specific privacy requirements must be independently assessed by qualified counsel. CEYO's non-custodial architecture means the deploying institution retains control of artifact signing and storage; CEYO's privacy properties are therefore strongest when the operator implements them fully within their own infrastructure. NIST SP 800-188 and ISO/IEC 29101 inform this framework but do not constitute certification. CEYO provides the record. Institutions provide the oversight.
