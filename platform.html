How NADIR produces an evidence package

NADIR sits beside inference and emits structured artifacts for retention and review. It is designed to integrate without requiring model weight disclosure.

Flow

  1. Inference request enters the runtime (prompt + policy context + inputs).
  2. NADIR captures a canonicalized record of the run under configured conditions.
  3. Artifacts are hashed into a tamper-evident structure (chain-of-custody ready).
  4. Verification fields are emitted for auditors without requiring proprietary disclosure.
  5. Artifacts are stored per retention policy and can be replay-verified for disputes.

Outputs

  • Run manifest (inputs, policy IDs, runtime identifiers, timestamps)
  • Deterministic transcript fields (canonicalized prompt/response payloads)
  • Integrity chain (hashes, signatures, verification fields)
  • Event timeline (system + model events, failures, escalations)
  • Disclosure profile (what is included / withheld by level)

Disclosure levels

Level I — Public abstract

High-level description of artifact types and verification method.

Level II — Controlled overview

Interface details, sample artifacts, and workflow diagrams under NDA.

Level III — Technical review

Implementation specifics, threat model, and validation plan with redactions.

Level IV — Full evaluation

Hands-on test plan and artifact validation in a controlled environment.

Non-claims

NADIR does not certify model safety, correctness, or “truth.” It supports evidentiary review by providing reproducible artifacts and tamper-evident integrity suitable for oversight and dispute handling.
