Architecture

NADIR follows a strict mechanics chain: attach at inference boundary, capture policy-scoped fields, normalize to canonical artifact, canonicalize deterministically, hash and digitally sign, apply disclosure-tier enforcement, and enable independent verification.

Operational integration Document suite

Stage 1 — Capture model

NADIR attaches non-intrusively via sidecar process, gateway middleware, or API wrapper. Instrumentation occurs outside core model weights and inference logic.

Policy-scoped capture
Capture scope is defined by configuration policy, specifying included fields, excluded fields, masking rules, disclosure tier, and version identifier.
Excluded by default
Model weights · Architecture internals · Proprietary training data · Unrelated system metadata
Mandatory fail-open architecture
If sealing fails: inference continues uninterrupted, failure event is logged, artifact is marked unsealed, and availability is preserved. The artifact layer is designed to avoid interrupting mission-critical inference pathways.

Stage 2 — Canonical artifact schema

Captured context is normalized into a canonical artifact structure suitable for deterministic hashing. The schema is designed for audit workflows and chain-of-custody-compatible storage.

Canonical artifact fields (example)

Timestamp · Provenance metadata · Input hash · Policy ID · Policy version · Policy hash · Capture environment fingerprint · Canonicalization version · Disclosure tier · Integrity fields · Verification fields

Stage 3 — Integrity model

Artifacts are sealed using cryptographic hashing and digital signatures. JSON artifacts are canonicalized using RFC 8785 prior to hashing. Deterministic serialization ensures consistent hash computation across verification environments.

Hashing

Reference implementation may include SHA-256. Alternative cryptographic suites may be selected based on regulatory or operational requirements.

Digital signature

Reference implementation may include ECDSA (P-256). Keys are managed according to deployment configuration.

Time anchoring
Optional by deployment. May include RFC 3161 trusted timestamp authority integration, hardware security module time source, or cryptographically verifiable time attestation. Deployment-defined and roadmap-phase dependent.
Note
Cryptographic selections are reference implementations and may evolve as standards and regulatory requirements change.

Stage 4 — Disclosure field model

NADIR enforces tiered visibility. Disclosure boundaries are policy-defined and configurable per deployment.

Public / redacted tier

Artifact summary, Policy ID, and non-sensitive metadata only.

Controlled review tier

Expanded fields, integrity verification data, and policy-aligned contextual elements for authorized reviewers.

Stage 5 — Verification workflow

Verification performs deterministic canonicalization (RFC 8785), hash recomputation, digital signature validation, and policy alignment check.

Does

  • Confirm artifact integrity
  • Confirm artifact provenance
  • Confirm policy conformity

Does not

  • Confirm model correctness
  • Confirm decision legality
  • Guarantee semantic equivalence of outputs
  • Represent regulatory sufficiency
Boundary declaration
  • NADIR verifies artifact integrity and provenance under defined policy.
  • NADIR does not certify compliance, legality, or correctness.
  • NADIR does not guarantee inference reproducibility.
  • Oversight conclusions are made by authorized institutions.
  • Admissibility of artifacts is determined by relevant authorities.
