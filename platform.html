Platform · Components · Interfaces

Deploy a layer, not a rewrite.

CEYO is organized as a small set of deterministic components: policy scope, canonicalization, sealing, disclosure tiers, and verification. Integration surfaces are designed to fit existing inference and governance stacks.

View components→ Interface tiles→

Core components

Click any component for a compact “what it does” and “what it does not do.”

Policy scope

Capture policy engine

Defines inclusion, exclusion, masking, and disclosure tiers before artifact generation.

What it does: makes scope explicit and versioned.
What it does not do: decide compliance outcomes.
Canonicalization

Deterministic canonicalizer

Normalizes captured fields into a stable schema and produces deterministic bytes (RFC 8785).

What it does: enables reproducible hashing across environments.
What it does not do: guarantee inference reproducibility.
Integrity sealing

Hash + signature module

Cryptographically seals canonical artifacts to make tampering detectable and provenance verifiable.

What it does: tamper evidence + origin authenticity under declared custody.
What it does not do: adjudicate decision correctness.
Disclosure

Tiered disclosure controls

Supports redacted and controlled-review views without post hoc improvisation.

What it does: preserves trade-secret boundaries while enabling verification.
What it does not do: expand scope beyond policy.
Verification

Verifier toolkit

Independent verification flow: canonicalize → hash → signature → policy alignment.

What it does: produces PASS / FAIL / POLICY_MISMATCH states.
What it does not do: determine legal sufficiency.
Retention

Artifact retention posture (optional)

Storage and chain-of-custody handling are operator-defined; CEYO supports audit-ready structure.

What it does: supports retrieval logging and review workflows.
What it does not do: enforce institutional custody procedures.

Interface tiles

Small, readable surfaces. No long blocks.

Sidecar / proxy attachment Deployment

Mirror policy-scoped fields at the inference boundary without mutating request/response.

mode: "sidecar" fail_open: true capture_policy: "POL-001@V1"
Artifact output Record

Canonical payload + integrity fields + policy identifiers.

artifact_id: "ART-…" policy_id: "POL-001" canonicalization: "RFC8785" digest: "…" signature: "…"
Verifier invocation Verify

Independent validation produces deterministic outcomes.

verify(artifact): canonicalize() recompute_hash() validate_signature() check_policy()
Disclosure tiers Scope

Views are determined at generation time, not after-the-fact redaction.

tier: "public_redacted" tier: "controlled_review" masking_rules: ["…"]

Inputs and outputs

A simple mental model for what CEYO sees and what it produces.

Inputs
  • Inference request/response context (policy-scoped fields only)
  • Policy ID, version, and masking rules
  • Operator key custody environment (KMS/HSM/TEE reference)
Outputs
  • Canonical artifact payload (deterministic representation)
  • Hash digest + signature for integrity and provenance
  • Verification states: PASS / FAIL / POLICY_MISMATCH
