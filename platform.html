[Architecture summary]

Attach Without Mutating Inference Authority

CEYO is designed as an external evidentiary substrate. It does not modify model weights. It does not intervene in decision logic. It produces records under declared policy scope for later verification.

Non-mutating
Interception and artifact generation occur without modifying request or response payloads.
Deterministic
Logically equivalent artifacts produce identical canonical bytes and identical digests under the same policy version.
Non-custodial
Private signing keys remain within operator-controlled HSM or KMS environments. CEYO does not store private keys.
Tiered disclosure
Artifacts support disclosure tiers so verification can occur without default exposure of proprietary internals.
Mechanics chain Attach → Capture → Canonicalize → Seal → Disclose → Verify
01
Attach
Sidecar, gateway, or wrapper placement at the inference boundary.
02
Capture
Policy-scoped fields only. Inclusion, exclusion, masking defined pre-capture.
03
Canonicalize
Deterministic normalization. RFC 8785 JSON canonicalization for stable bytes.
04
Seal
Hash and signature applied to canonical bytes for tamper-evidence and origin authenticity.
05
Disclose
Tiered fields by authorization. Scope is explicit, not implied.
06
Verify
Recompute canonicalization and hash. Validate signature. Confirm policy alignment.
[Attachment patterns]

Inference Boundary Integration

CEYO integrates at the inference boundary through one of three operator-selected patterns. The inference engine remains unchanged. The evidentiary layer operates externally.

Deployment
Sidecar
Co-located process beside the inference service. Minimal pipeline disruption. Suitable for containerized environments.
Deployment
Gateway
Intermediate layer between calling system and inference endpoint. Suitable for API-based inference architectures.
Deployment
Wrapper
Service wrapper around inference calls. Suitable when interface compatibility and routing control are required.

Separation rule: Attachment must not require mutation of model weights, training pipelines, or inference engines. CEYO remains external to model internals by design.

[Canonical artifact]

Deterministic Structure, Then Sealing

Artifact generation begins with policy-defined capture. Masking and exclusions occur before canonicalization. Canonical bytes are produced deterministically and sealed for later verification.

Canonical artifact header — minimal example illustrative
{
  "artifact_id": "c2e1c8d0-9c6b-4d2a-9c4a-8d3f7d1d1b31",
  "timestamp": "2026-02-19T17:14:22Z",
  "policy_id": "POL-CEYO-001",
  "policy_version": "1.0.0",
  "disclosure_tier": "redacted",
  "canonicalization": {
    "standard": "RFC8785",
    "version": "1"
  },
  "integrity": {
    "hash_algorithm": "SHA-256",
    "signature_algorithm": "ECDSA-P256",
    "digest": "base64url(sha256(canonical_bytes))",
    "signature": "base64url(signature_over_digest)",
    "public_key_reference": "kms:key/alias/ceyo-seal"
  }
}
Verification recomputes canonicalization and hash deterministically, validates the signature, and confirms policy alignment. Verification confirms integrity and provenance under policy scope — not correctness of model output.

Ordering requirement: Redaction → Canonicalization → Deterministic byte stream → Hash → Signature. Integrity is computed over the post-redaction canonical form.

[Operational discipline]

Fail-Open by Default

Integrity infrastructure must not become a single point of failure. In fail-open posture, inference continues even if artifact sealing fails. Failures are logged and artifacts are explicitly marked as unsealed and reviewable.

Fail-open
Proxy-layer enforcement. A crash, timeout, or KMS failure must not block inference execution.
Determinism
Canonical bytes and digests must match across instances and environments under the same policy version.
Boundary Declaration  ·  V2026.02.19

CEYO does not modify model weights, architecture, or inference outputs.

CEYO does not evaluate correctness, fairness, legality, or compliance outcomes.

CEYO does not guarantee inference reproducibility.

CEYO supports constrained disclosure; scope is policy-defined before capture.

Private signing keys remain operator-custodied within HSM or KMS environments.

Authorized institutions and operators retain responsibility for determinations.

Verification confirms integrity and provenance under declared policy scope. It does not confirm correctness of model output.
