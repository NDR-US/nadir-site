Inference Boundary Attachment

CEYO integrates at the inference boundary through a sidecar, gateway, or API wrapper configuration. This posture is intentional. Evidence infrastructure should not require mutation of model weights, training pipelines, or core inference engines.

Attachment at the boundary allows CEYO to:

  • Observe defined inference events
  • Capture policy-scoped context
  • Normalize event structure
  • Produce deterministic artifacts
  • Seal artifacts without interfering with decision flow
Excluded by default
  • Model weights
  • Training datasets
  • Proprietary architecture internals
  • Unrelated system telemetry
  • Full conversational transcripts unless explicitly configured

Policy-Scoped Capture

Artifact generation begins with explicit policy definition. The deploying institution defines capture scope before any data is collected.

  • Which fields are included
  • Which fields are excluded
  • Which fields are masked
  • Disclosure tier configuration
  • Policy version identifiers

Capture scope is versioned and recorded within the artifact itself. Reviewers can verify not only integrity, but the declared capture boundary under which the artifact was produced.

Sealing

Deterministic Canonicalization & Cryptographic Sealing

Before hashing, JSON artifacts are canonicalized using RFC 8785 (JSON Canonicalization Scheme). Canonicalization ensures that logically equivalent data structures produce identical byte representations across environments.

CEYO treats canonicalization as a structural requirement, not an optimization. Without deterministic canonicalization, verification becomes environment-dependent.

After canonicalization, artifacts are sealed using cryptographic hashing and digital signatures. Reference implementations: SHA-256 hashing · ECDSA P-256 signatures. Suite selection is configurable based on regulatory and operational requirements.

Signing keys are generated and retained within the client-controlled environment — HSM, KMS, or TEE. CEYO does not possess, transmit, or store private signing keys. This non-custodial architecture preserves a clear separation between infrastructure provider and cryptographic authority.

Integrity–Availability Balance

Integrity infrastructure must not compromise operational continuity by default. CEYO's fail-open design ensures that inference remains available even when sealing encounters a failure condition.

Fail-open behavior
  • Inference may continue uninterrupted
  • The failure event is logged
  • The artifact is marked as unsealed
  • The condition becomes reviewable

Fail-open behavior preserves availability unless the operator explicitly configures a stricter posture. Operational control remains with the deploying institution.

Disclosure Tier Enforcement

Artifacts support tiered disclosure. Disclosure boundaries are policy-defined before artifact generation. Reviewers verify within declared scope rather than relying on post hoc redaction.

Disclosure tiers
Public / Redacted

Boundary Declaration  ·  V2026.02.19

CEYO verifies integrity and provenance under defined policy; it does not certify compliance, legality, or correctness.

CEYO does not guarantee inference reproducibility or semantic equivalence of outputs.

Admissibility of artifacts is determined by the relevant authority, not by CEYO.

Cryptographic suite references are illustrative; deployments select suites appropriate to their regulatory context.

It provides deterministic artifacts. Institutions determine their meaning.
